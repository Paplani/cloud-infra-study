# Security Group 점검 및 실습

## 1. 실습 목적

AWS Security Group이 EC2 인스턴스의 외부 접속에 어떤 영향을 주는지 확인했다.

이번 실습의 목적은  
단순히 포트를 여닫는 것이 아니라,  
**네트워크 접근 제어가 실제 서비스 접속에 어떻게 작동하는지 이해하는 것**이다.

특히 아래 내용을 중점적으로 확인했다.

- SSH(22) 접속 제어
- HTTP(80) 접속 제어
- 포트 변경에 따른 접속 성공/실패 변화
- 서비스가 정상이어도 포트가 막히면 접속이 안 된다는 점

---

## 2. 실습 환경

- Cloud: AWS EC2
- OS: Ubuntu
- Web Server: nginx
- Access:
  - SSH
  - HTTP (브라우저)

---

## 3. 실습 목표

- Security Group의 역할 이해
- 인바운드 규칙과 외부 접속의 관계 확인
- 22번 / 80번 포트의 의미 이해
- 포트 차단 시 접속 실패 확인
- 포트 복구 시 정상 동작 확인

---

## 4. Security Group 개념 정리

Security Group은 EC2에 적용되는 **가상 방화벽**이다.

서버와 nginx가 정상 실행 중이어도  
필요한 포트가 열려 있지 않으면 외부에서는 접속할 수 없다.

### 주요 포트

- SSH (22/tcp) → 서버 원격 접속
- HTTP (80/tcp) → 웹 브라우저 접속

즉,  
**서비스 상태와 네트워크 접근 정책은 별개의 요소**이다.

---

## 5. 작업 과정

### 5-1. 현재 상태 확인

Security Group 인바운드 규칙:

- SSH 22 허용
- HTTP 80 허용

브라우저 접속:

```text
http://<public-ip>
```

nginx 페이지가 정상적으로 열렸다.

이 상태는 다음 조건이 모두 만족된 상태이다.

- EC2 실행 중
- nginx 정상 실행
- 80 포트 열림
- Security Group 허용

---

### 5-2. HTTP(80) 규칙 제거

80번 포트를 제거한 후 다시 접속했다.

결과:

- 웹 접속 실패 (timeout)

👉 nginx는 정상인데 접속이 안 됐다  
→ **문제는 서버가 아니라 네트워크**

---

### 5-3. HTTP 복구

80번 포트를 다시 추가하니  
웹 접속이 정상으로 돌아왔다.

---

### 5-4. SSH(22) 규칙 점검

SSH 접속 확인:

```bash
ssh -i my-key.pem ubuntu@<public-ip>
```

또한 SSH는 전체 공개보다  
**내 IP만 허용하는 것이 더 안전하다는 점을 이해했다.**

---

## 6. 확인 결과

- HTTP 차단 → 웹 접속 불가
- HTTP 허용 → 웹 접속 가능
- 포트 설정이 실제 접속 여부에 직접 영향

즉,  
**서비스 상태 + 네트워크 설정이 모두 맞아야 접속 가능하다**

---

## 7. 운영 관점에서 중요한 이유

실제 운영에서는  
“웹이 안 열린다”, “SSH가 안 된다”는 문제가 자주 발생한다.

이때 원인은 다양하다.

- 서버 문제 (nginx)
- 네트워크 문제 (Security Group)
- 설정 오류 (IP, 포트 등)

이번 실습을 통해  
**Security Group도 하나의 독립적인 장애 원인이 될 수 있다는 점을 확인했다.**

따라서 접속 문제 발생 시  
**서버 + 네트워크를 함께 확인하는 것이 중요하다.**

---

## 8. 발생할 수 있는 문제

### 웹 접속 실패

가능한 원인:

- 80 포트 차단
- nginx 중지
- 퍼블릭 IP 오류

---

### SSH 접속 실패

가능한 원인:

- 22 포트 차단
- pem 파일 문제
- 사용자명 오류

---

## 9. 트러블슈팅 체크리스트

### 웹 접속이 안 될 때

아래 순서대로 확인한다.

1. 인스턴스 Running 상태 확인  
2. 퍼블릭 IP 확인  
3. Security Group에서 80 포트 확인  
4. nginx 상태 확인  

```bash
sudo systemctl status nginx
```

5. 포트 리스닝 확인  

```bash
sudo ss -tulnp | grep :80
```

👉 이 순서로 보면  
서버 문제인지 네트워크 문제인지 구분할 수 있다.

---

### SSH 접속이 안 될 때

1. 인스턴스 Running 상태 확인  
2. Security Group에서 22 포트 확인  
3. 접속 명령어 확인  

```bash
ssh -i my-key.pem ubuntu@<public-ip>
```

4. pem 파일 권한 확인  

```bash
chmod 400 my-key.pem
```

5. 사용자명 확인 (ubuntu)

---

👉 핵심

문제는 감으로 해결하는 게 아니라  
**순서대로 확인하면서 좁혀가는 것이 중요하다.**

---

## 10. 배운 점

- Security Group은 EC2 접근을 제어하는 핵심 요소다
- 포트 설정 하나로 서비스 접근 여부가 달라진다
- 접속 문제는 서버와 네트워크를 구분해서 봐야 한다
